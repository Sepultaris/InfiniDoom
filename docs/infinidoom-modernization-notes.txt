InfiniDoom modernization notes
==============================

InfiniDoom is a Zandronum-derived engine modernization effort. UZDoom and
GZDoom are useful references for expected behavior, bug fixes, and mature
subsystem designs, but they are not the target architecture by default.

Guidelines:

- Preserve Zandronum multiplayer compatibility and user-facing behavior unless
  a change is explicitly intended to break from it.
- Prefer small, reviewable modernization steps over wholesale subsystem
  replacement.
- Use UZDoom/GZDoom as behavioral references first. Direct backports should be
  identified and documented with the upstream source when possible.
- Keep new backends shaped around the interfaces this tree already exposes
  unless replacing that interface is the explicit task.
- Build and test each backend independently so FMOD-era behavior remains easy
  to compare against OpenAL, FluidSynth, and future Vulkan work.

Current audio direction:

- FMOD remains the legacy/reference backend.
- OpenAL is an incremental native backend for sound effects and streaming
  music, not a copy of UZDoom's larger OpenAL implementation.
- FluidSynth support is used for long-term MIDI playback through the existing
  MIDI device abstraction.
- UZDoom's audio code can inform future improvements such as source pooling,
  decoded SFX support, and EFX/reverb, but those should be integrated in
  InfiniDoom-sized steps.

Current Vulkan direction:

- The Vulkan renderer starts as a runtime bootstrap selected by vid_renderer 2.
  It dynamically loads the Vulkan loader, creates an instance and Win32 surface,
  finds a physical device with graphics/present support, and creates a logical
  device with VK_KHR_swapchain enabled. The Win32 path also creates a swapchain
  and records its image count, format, and extent. Command buffer, image view,
  and synchronization setup exists. The first Win32 Vulkan framebuffer path uses
  the software renderer's paletted buffer and presents it through Vulkan during
  screen->Update().
- The Vulkan renderer should not present from renderer initialization because
  that can obscure the framebuffer path. Presentation belongs in the video /
  framebuffer backend.
- Renderer diagnostics are exposed through the existing stat overlay system as
  "stat renderer". The first pass reports selected renderer, engine version,
  framebuffer mode, process RAM, and Vulkan device/swapchain/upload-buffer
  data. Vulkan now reports real GPU presentation time through timestamp queries
  when supported and live VRAM usage/budget through VK_EXT_memory_budget when
  the driver exposes it.
- The Vulkan window path should always treat the Win32 client size as the
  swapchain authority. Logical software framebuffer size can differ from the
  swapchain in borderless fullscreen, so presentation must scale into the
  current swapchain extent and recreate when the client extent changes.
- The first GPU presentation path uploads the existing software renderer's
  8-bit framebuffer and active palette to sampled Vulkan images. A fullscreen
  triangle shader performs palette lookup and scaling into the swapchain. The
  previous CPU-expanded transfer copy remains as a fallback while the GPU path
  matures.
- `vk_present_filter` controls Vulkan palette presentation filtering. The
  default value `0` keeps nearest filtering for the classic pixel look; value
  `1` enables the old experimental linear sampling of palette indices; value
  `2` keeps nearest index sampling and blends the final palette colors in the
  shader for a more visible, less color-corrupting smoothing experiment.
- `vk_present_sharpness` tunes the `vk_present_filter 2` shader from smoother
  color blending at `0` to crisper pixel-edge preservation at `1`.
- `vk_render_scale` currently controls the Vulkan presentation source scale for
  quality testing. Values below `1.0` downsample the completed software frame
  before uploading it to Vulkan, then the palette presenter upscales it to the
  swapchain. This proves and tunes the presentation/upscale path; a later pass
  should separate true internal render resolution from logical UI and output
  resolution.
- `vk_scene_probe` is an opt-in Vulkan geometry proof. It uses a translucent
  debug-marker pipeline, fills a host-visible vertex buffer with 3D Doom-world
  positions near the current camera, pushes a world-to-clip transform to the
  vertex shader, and issues a draw call after the palette presenter.
- `vk_world_probe` shares the debug vertex stream but uses its own opaque,
  depth-tested pipeline. It fills the vertex buffer from nearby one-sided Doom
  map lines, emitting simple wall quads as the first backend-owned batch of
  actual level geometry. Wall segments are clipped against a small camera near
  plane and the left/right view-frustum planes, which keeps this probe from
  drawing behind-camera or offscreen slabs across the foreground. The Vulkan
  probe/world extraction path and the shared world-to-clip transform currently
  default to a 180-degree-rotated negated Doom view angle. Its side vector is
  the opposite perpendicular of that forward vector, matching Doom's lateral
  movement handedness for strafe tests.
- `vk_draw_world` is the first non-probe Vulkan world geometry toggle. It
  reuses the clipped one-sided wall extraction but writes into a separate draw
  range and appears in `stat renderer` as `GPU world draw`. This is still
  flat-colored scaffolding; texture upload, UVs, and material binding are the
  next renderer milestone.
- Temporary calibration CVars `vk_world_yaw_sign` and `vk_clip_yaw_sign` are
  available while matching Doom's software-space camera basis. Once the correct
  combination is verified in-game, collapse these back into fixed renderer
  math. `vk_clip_side_sign` was intentionally removed after it could crash the
  diagnostic renderer path when changed live.
- The Vulkan presentation render pass now owns a depth attachment. The palette
  presenter ignores depth, while the world-probe geometry pipeline depth-tests
  and writes camera-distance depth values.
- `vk_present_scale_mode` controls how the Vulkan palette presenter maps the
  software framebuffer into the swapchain: `0` stretches to the full client
  size, `1` preserves aspect, and `2` uses integer scaling against a 320-wide
  virtual canvas. `vk_present_aspect` defaults to `0`, which means use the
  source framebuffer aspect. Explicit aspect overrides require
  `vk_present_force_aspect 1` and are experimental because true classic 4:3
  must be applied before the frame/menu is composed.
- Next steps should be:
  1. add SDL surface creation for non-Windows targets;
  2. harden resize, alt-tab, and swapchain recreation;
  3. add higher-quality scaling/color conversion shaders around the palette
     presenter;
  4. move renderer-owned scene features to GPU paths incrementally.

Release discipline:

- Every checkpoint build should update CHANGELOG.md with user-visible behavior,
  backend changes, and known limitations.
- Version strings in src/version.h should be bumped for each committed vDoom
  checkpoint build. Network, demo, and save versions should only be bumped when
  those formats or compatibility contracts actually change.
- Build before committing. For the current Windows development tree, the main
  verification targets are:
  - cmake --build build\vs2022-vcpkg-openal-fluid --config Debug --target zdoom -- /m
  - cmake --build build\vs2022-vcpkg-openal-fluid --config Debug --target pk3 -- /m
